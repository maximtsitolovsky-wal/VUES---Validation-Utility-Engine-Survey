Architecture
Guide

Fully automated QA pipeline for retail site survey submissions. Records arrive in Airtable as QUEUED, flow through 7 stages, then PASS or FAIL — vendor emailed, Airtable updated, everything archived, lessons fed back into memory. No human needed in the happy path.

● Stage 1 — Entry Point

▣ main.py core

The system’s heartbeat. Polls Airtable for QUEUED records on a 60-second loop. Catches all exceptions at the record level so one bad submission can never crash the pipeline. Spawns the Async Worker thread on startup. Handles OS shutdown signals gracefully. If this stops, nothing else runs.

● Stage 2 — Validate & Recall

▣ Validate & Recall phase

Before touching any files, the pipeline checks that every critical field exists: Site Number, Site Name, attachments, vendor info. A missing field immediately marks the record ERROR and halts — expensive processing is never wasted on incomplete submissions.

▣ memory.py memory

Simultaneously, memory.py queries the lesson archive by site, vendor, and tag. Any previously extracted failure rules (e.g. “Vendor X always drops column Y”) are surfaced here and attached to the grading context, making each run smarter than the last.

● Stage 3 — Process

▣ Download Attachment

airtable_client.py downloads the XLSX or CSV attachment to a local temp/ folder. RISK-003 — Airtable CDN URL expiry — is mitigated: a mirror script backs up attachments to OneDrive by Site ID.

▣ Normalize (file_processor.py)

Headers are lowercased and underscored regardless of how the vendor formatted them. Critically, project_id is always overwritten with the Airtable Site Number — the single source of truth. Vendor-supplied project IDs are never trusted.

▣ SQL Reference Fetch (sql.py)

Pulls the canonical reference row set from local SQLite, scoped strictly to the current site. This is the baseline the grader compares against. All reads are read-only. sql.py owns all database interactions — nothing else touches the DB.

● Stage 4 — Grade & Decide

▣ python_grader.py

Compares the normalized submission against the SQL reference row-by-row. Uses column fingerprinting and weighted scoring to produce a numeric score 0–100. Memory recall context (from Stage 2) can inform the summary narrative.

▣ PASS branch ≥ threshold

status_from_score() is the single decision function — never duplicated. A PASS routes through Output (email + Airtable update) then Archive. If score ≥ 95, post_pass_correction also runs to generate corrected reference files.

▣ FAIL branch < threshold

Routes through the same Output stage (vendor receives a FAIL email with specific feedback), then through Archive where a lesson is extracted. That lesson flows into the learning loop and is available for the next run against this site or vendor.

● Stage 5 — Output

▣ emailer.py

Sends a templated email (PASS / FAIL / ERROR) to the vendor. SMTP is optional — if SMTP_ENABLED=false, the step is silently skipped. Email content includes score, summary, and corrective guidance for failures.

▣ Airtable Update (airtable_client.py)

PATCHes the Airtable record with status, True Score, Score%, Fail Summary, and Notes. All fields are always written — no field is ever left blank from a previous run. This is a hard rule preventing ghost data contamination across resubmissions.

● Stage 6 — Archive & Learn

▣ archive.py immutable

Appends an execution record to a JSON archive and stores the raw submission file. Nothing is ever deleted or overwritten. This provides a full audit trail for every run, including errors.

▣ post_pass_correction score ≥95 only

For very high-scoring passes, generates corrected RAW, CSV, and LOG output files. Never re-grades. Never calls Airtable. A completely separate, safe sub-pipeline.

▣ reviewer.py audit

Internal static audit of the run’s behaviour. Issues are classified CRITICAL / HIGH / LOW. Results are logged and archived. reviewer.py never mutates pipeline state — it is a pure observer.

● The Learning Loop

▣ lesson → memory feedback

After a failure (or even a partial pass), the pipeline extracts a generalized rule with a confidence score (default 0.75). This rule is tagged by site, vendor, column pattern, and issue type, then written to the archive. On the next run, memory.py recalls it during Stage 2 — before grading even starts — so the grader has institutional context. Over time, this builds a compounding knowledge base of failure patterns.

● Stage 7 — Async Workers

▣ MetricsRefreshWorker

Runs on a background thread. mark_dirty() is called whenever a record completes. The worker detects the dirty flag and asynchronously recomputes 3 metric CSVs and refreshes 2 HTML dashboards — never blocking the main poll loop. If metrics fail, the pipeline keeps running.
